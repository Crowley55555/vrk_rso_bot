# vrk_rso_bot

Telegram-бот и бот для мессенджера **Max** для управления задачами и авариями с хранением данных в **Google Sheets**. Доступ к таблице выполняет отдельный **FastAPI**-сервис; оба бота ходят в него по HTTP с заголовком `X-API-Key`.

Бот поддерживает две роли:
- обычный пользователь: может добавить задачу;
- администратор: может добавлять задачи, просматривать списки, редактировать задачи, брать их в работу и переносить в выполненные.

## Что умеет бот

- добавлять новые задачи в лист `Не начатые`;
- показывать задачи из листов `Не начатые`, `В работе`, `Выполненные`;
- открывать карточку задачи с кнопками действий;
- редактировать комментарии, срок выполнения и (для листа «В работе») ответственных;
- переносить задачу из `Не начатые` в `В работе` («Взять в работу»);
- переносить задачу из `В работе` в `Выполненные` («Отметить как выполненное»);
- сразу отмечать задачу как выполненную с листа `Не начатые` (прямой перенос в архив, минуя «В работе»);
- удалять служебные сообщения в рамках сессии;
- работать с Google Sheets через Service Account.

## Стек проекта

- Python 3.11+
- **API:** FastAPI, uvicorn, gspread, pydantic
- **Telegram:** python-telegram-bot v20+, httpx
- **Max:** httpx, long polling `GET /updates`
- `python-dotenv`

## Структура Google Sheets

Используется таблица:

- `SPREADSHEET_ID=1MLdHFoSgTA8nujYeJnqZzc6BdGDtnbQRzUPJ2cRZ2UY`

В таблице должно быть три листа:

### 1. `Не начатые`

Столбцы:
- `A` = `Дата добавления`
- `B` = `Наименование задачи`
- `C` = `Комментарии`
- `D` = `Ответственные`
- `E` = `Срок выполнения`
- `F` = `Кто добавил`

### 2. `В работе`

Столбцы:
- `A` = `Дата`
- `B` = `Наименование задачи`
- `C` = `Комментарии`
- `D` = `Ответственные`
- `E` = `Срок`
- `F` = `Кто добавил`

### 3. `Выполненные`

Столбцы:
- `A` = `Дата`
- `B` = `Наименование задачи`
- `C` = `Комментарии`
- `D` = `Ответственные`
- `E` = пусто или `Срок`
- `F` = `Кто добавил`

Важно:
- первая строка каждого листа должна быть строкой заголовков;
- данные должны начинаться со второй строки;
- имена листов должны совпадать точно, включая пробелы и регистр.

## Что такое `credentials.json` и зачем он нужен

`credentials.json` — это JSON-файл сервисного аккаунта Google Cloud, через который бот получает доступ к Google Sheets API.

Проще говоря:
- Telegram-бот сам по себе не умеет читать и записывать Google Sheets;
- чтобы работать с таблицей, ему нужно пройти аутентификацию в Google;
- для серверных приложений и ботов для этого обычно используют `Service Account`;
- `credentials.json` содержит технические учетные данные этого сервисного аккаунта.

Внутри такого файла находятся:
- email сервисного аккаунта;
- `private_key`;
- `client_id`;
- служебные URI Google API.

Без `credentials.json` бот не сможет:
- открыть таблицу;
- прочитать список задач;
- добавить строку;
- обновить ячейку;
- удалить или перенести задачу.

## Важно по безопасности

Файл `credentials.json`:
- нельзя публиковать в открытый доступ;
- нельзя коммитить в публичный репозиторий;
- нельзя пересылать посторонним;
- желательно хранить только локально или в защищенном хранилище секретов.

Если этот файл утечет, посторонний сможет использовать ваш сервисный аккаунт для доступа к Google API в рамках выданных прав.

## Как получить `credentials.json`

### Шаг 1. Создайте проект в Google Cloud

1. Перейдите в [Google Cloud Console](https://console.cloud.google.com/).
2. Создайте новый проект или выберите существующий.

### Шаг 2. Включите Google Sheets API

1. Откройте раздел `APIs & Services`.
2. Перейдите в `Library`.
3. Найдите `Google Sheets API`.
4. Нажмите `Enable`.

При желании можно также включить `Google Drive API`, хотя для текущего сценария основной нужен именно `Google Sheets API`.

### Шаг 3. Создайте Service Account

1. Откройте `APIs & Services` -> `Credentials`.
2. Нажмите `Create Credentials`.
3. Выберите `Service account`.
4. Укажите имя, например `vrk-rso-bot`.
5. Завершите создание.

### Шаг 4. Скачайте JSON-ключ

1. Откройте созданный сервисный аккаунт.
2. Перейдите во вкладку `Keys`.
3. Нажмите `Add Key` -> `Create new key`.
4. Выберите формат `JSON`.
5. Скачанный файл сохраните в корень проекта под именем `credentials.json`.

Итоговая структура проекта будет выглядеть так:

```text
vrk_rso_bot/
├── .env
├── .env.example
├── credentials.json
├── docker-compose.yml
├── api/
├── shared/
├── telegram_bot/
├── max_bot/
├── requirements.txt
└── README.md
```

## Как выдать доступ сервисному аккаунту к таблице

После создания `credentials.json` нужно обязательно открыть этот файл и найти поле:

```json
"client_email": "example-name@project-id.iam.gserviceaccount.com"
```

Дальше:

1. Скопируйте значение `client_email`.
2. Откройте нужную Google Sheets таблицу.
3. Нажмите `Поделиться`.
4. Добавьте этот email как пользователя.
5. Выдайте права `Редактор`.

Если этого не сделать, бот не сможет работать с таблицей, даже если `credentials.json` лежит в проекте.

## Локальный запуск

### 1\. Установка зависимостей

Одной командой из корня репозитория (все сервисы — удобно для тестов):

```powershell
py -3 -m pip install -r requirements.txt
```

Если `pip` долго висит на `pypi.org` и пишет `Read timed out`, увеличьте таймаут и число повторов (часто помогает при медленном или нестабильном интернете):

```powershell
py -3 -m pip install -r requirements.txt --default-timeout=120 --retries 10
```

Либо один раз настроить зеркало PyPI ближе к вам (пример — зеркало Яндекса) и снова выполнить установку:

```powershell
py -3 -m pip install -r requirements.txt -i https://mirror.yandex.ru/mirrors/pypi/simple/ --trusted-host mirror.yandex.ru
```

Либо только нужные части:

```powershell
py -3 -m pip install -r api/requirements.txt
py -3 -m pip install -r telegram_bot/requirements.txt
py -3 -m pip install -r max_bot/requirements.txt
```

### 2\. Проверка настроек

- в корне проекта есть `.env` (шаблон — [`.env.example`](.env.example));
- для API доступны `credentials.json` и переменные `SPREADSHEET_ID`, `API_KEY`, `GOOGLE_CREDENTIALS_FILE` (локально часто `GOOGLE_CREDENTIALS_FILE=credentials.json`);
- у ботов в `.env` заданы `API_BASE_URL` и тот же `API_KEY`, что и у API;
- сервисному аккаунту Google выдан доступ к таблице (см. раздел выше).

### 3\. Запуск процессов локально

Рабочий каталог — **корень репозитория**.

**Терминал 1 — API:** для импорта пакета `api` задайте `PYTHONPATH` на корень репозитория.

```powershell
cd путь\к\vrk_rso_bot
$env:PYTHONPATH = "."
py -3 -m uvicorn api.main:app --reload --host 127.0.0.1 --port 8000
```

Проверка: откройте в браузере `http://127.0.0.1:8000/health` — ожидается `{"status":"ok"}`.

**Терминал 2 — Telegram-бот** (после старта API). В `.env` укажите `API_BASE_URL=http://127.0.0.1:8000` (или `http://localhost:8000`). Скрипт [`run.py`](run.py) сам добавляет в `sys.path` корень репозитория и каталог `telegram_bot`.

```powershell
cd путь\к\vrk_rso_bot
py -3 run.py
```

Альтернатива: из корня с `PYTHONPATH=.;telegram_bot` выполнить `py -3 -m bot.main`.

**Терминал 3 — Max-бот** (опционально): те же `API_BASE_URL` и `API_KEY`, плюс `MAX_BOT_TOKEN` и `MAX_ADMIN_IDS`.

```powershell
cd путь\к\vrk_rso_bot
$env:PYTHONPATH = "."
py -3 -m max_bot.main
```

Остановка любого процесса: `Ctrl+C` в соответствующем терминале.

На **Linux / macOS** замените `$env:PYTHONPATH = "."` на `export PYTHONPATH=.`.

### 4\. Локальное тестирование

1. **Health API** — `GET http://127.0.0.1:8000/health`.

2. **Защищённый эндпоинт** (подставьте свой `API_KEY` из `.env`; в URL — имя листа из таблицы, ниже — лист `Не начатые`):

```bash
curl -s -H "X-API-Key: ВАШ_API_KEY" "http://127.0.0.1:8000/api/v1/sheets/tasks/%D0%9D%D0%B5%20%D0%BD%D0%B0%D1%87%D0%B0%D1%82%D1%8B%D0%B5"
```

Если ключ неверный — ответ `401`. При корректном доступе к таблице — JSON с полем `tasks`.

3. **Сквозной сценарий:** поднять API → запустить одного бота → в мессенджере выполнить `/start` и пройти простой сценарий (например, список задач или сообщение об аварии). Записи должны появляться в Google Sheets.

4. **Telegram:** не запускайте две копии бота с одним `BOT_TOKEN` (polling и Docker одновременно) — будет конфликт `getUpdates`.

5. **Без реальных мессенджеров** можно ограничиться проверкой API через `curl`/Postman и логами `uvicorn`.

## Как работает логика ролей

### Обычный пользователь

Доступно:
- `/start`
- кнопка `Добавить задачу`

Сценарий:
1. Вводит наименование задачи.
2. Вводит комментарий или `-`.
3. Вводит ФИО.
4. Вводит срок выполнения или `-`.

После этого задача записывается в лист `Не начатые`.

### Администратор

Если Telegram ID пользователя есть в `TELEGRAM_ADMIN_IDS` (или в `ADMIN_IDS` для обратной совместимости), в главном меню он увидит:

- `➕ Добавить задачу`
- `📋 Задачи к выполнению`
- `🔄 В работе`
- `✅ Выполненные задачи`

В карточке задачи администратор видит кнопки:

- **✏️ Редактировать** — изменить комментарий и срок (для задач из «В работе» ещё и ответственных).
- **▶️ Взять в работу** — только для задач из «Не начатые»; переносит задачу в лист «В работе».
- **✅ Отметить как выполненная** — только для задач из «Не начатые»; переносит задачу сразу в «Выполненные», минуя «В работе».
- **✅ Отметить как выполненное** — только для задач из «В работе»; переносит в «Выполненные».

При редактировании задачи из листа **«В работе»** запрашиваются три шага: новый комментарий, новый срок, новые ответственные (ввод `-` оставляет значение без изменений). Для задач из «Не начатые» и «Выполненные» — только комментарий и срок.

## Поведение бота

- На каждом шаге диалога бот удаляет предыдущее сообщение бота и сообщение пользователя.
- Кнопка `◀️ Назад` возвращает на предыдущий шаг сценария.
- Кнопка `🏠 Главное меню` (reply или inline) завершает текущий сценарий и возвращает в главное меню.
- После некоторых действий (например, «Задача отмечена как выполненная и перенесена в архив») показывается inline-кнопка «🏠 Главное меню» для возврата в меню.
- Команда `/cancel` отменяет любой активный диалог.
- Если задач больше 30, показываются только последние 30.

## Работа с Google Sheets

Доступ к таблице реализован в сервисе **`api`** (`api/services/sheets_service.py`): те же операции, что раньше в `sheets.py`, доступны по HTTP (`/api/v1/sheets/...`) с заголовком `X-API-Key`.

Боты вызывают их через **`shared/api_client.py`** (`APIClient` / те же имена функций, что у модульного API).

Для всех операций записи в API используется `asyncio.Lock()`, чтобы избежать гонок при одновременных действиях нескольких пользователей.

## Логирование

В проекте настроено стандартное логирование Python.

В логах можно увидеть:
- запуск приложения;
- ошибки Telegram API;
- ошибки Google Sheets API;
- внутренние исключения приложения.

## Частые проблемы

### 1. Ошибка `Не задана переменная окружения BOT_TOKEN`

Причина:
- не создан `.env`;
- файл `.env` лежит не в корне проекта;
- переменная заполнена с ошибкой.

Решение:
- проверьте наличие `.env`;
- проверьте имя переменной;
- проверьте, что в значении нет лишних кавычек и пробелов.

### 2. Ошибка `Файл сервисного аккаунта не найден`

Причина:
- отсутствует `credentials.json`;
- неверный путь в `GOOGLE_CREDENTIALS_FILE`.

Решение:
- положите JSON-файл в корень проекта;
- либо укажите правильный абсолютный или относительный путь.

### 3. Бот не видит таблицу

Причина:
- сервисному аккаунту не выдан доступ к таблице.

Решение:
- откройте `credentials.json`;
- скопируйте `client_email`;
- добавьте его в доступ к Google таблице с правами `Редактор`.

### 4. Ошибка при чтении или записи в Google Sheets

Причина:
- не включен `Google Sheets API`;
- неверная структура листов;
- изменены названия вкладок;
- отсутствуют обязательные заголовки.

Решение:
- проверьте API в Google Cloud;
- проверьте точные имена листов;
- проверьте заголовки в первой строке.

## Рекомендации по эксплуатации

- держите `.env` и `credentials.json` вне публичного репозитория;
- периодически проверяйте, что структура листов не изменилась;
- перед переносом на сервер настройте отдельный сервисный аккаунт для production;
- при необходимости можно завернуть запуск в `Task Scheduler`, `systemd` или Docker.

## Деплой на сервер (Docker)

Ниже — базовый сценарий развёртывания на любом Linux\-сервере с Docker и Docker Compose.

### 1\. Предварительные условия

На сервере должны быть установлены:

- `docker`
- `docker compose` (новая версия, не `docker-compose` в виде отдельного бинарника)

Проверка:

```bash
docker --version
docker compose version
```

### 2\. Подготовка файлов локально

Убедитесь, что локально в корне проекта есть:

- `.env` — см. шаблон [`.env.example`](.env.example): `BOT_TOKEN`, `MAX_BOT_TOKEN`, `API_KEY`, `API_BASE_URL`, `SPREADSHEET_ID`, `GOOGLE_CREDENTIALS_FILE`, `TELEGRAM_ADMIN_IDS`, `MAX_ADMIN_IDS`;
- `credentials.json` — ключ сервисного аккаунта (монтируется в контейнер **api**);
- `docker-compose.yml`.

Для Docker задайте путь к ключу для сервиса API, например:

```env
GOOGLE_CREDENTIALS_FILE=/app/credentials.json
```

### 3\. Копирование проекта на сервер

Вариант через `git`:

1. Создайте приватный репозиторий и запушьте код **без** `credentials.json` и `.env` (они уже в `.gitignore`).
2. На сервере:

```bash
git clone <url_вашего_репозитория> vrk_rso_bot
cd vrk_rso_bot
```

3. Скопируйте на сервер `credentials.json` и `.env` (по `scp`, SFTP или другим способом) в корень проекта:

```bash
scp credentials.json user@server:/path/to/vrk_rso_bot/credentials.json
scp .env user@server:/path/to/vrk_rso_bot/.env
```

Либо перенесите проект целиком архивом:

```bash
tar czf vrk_rso_bot.tar.gz vrk_rso_bot
scp vrk_rso_bot.tar.gz user@server:/path/to/
```

И на сервере:

```bash
cd /path/to
tar xzf vrk_rso_bot.tar.gz
cd vrk_rso_bot
```

### 4\. Сборка Docker\-образа

В корне проекта на сервере:

```bash
docker compose build
```

Команда собирает три образа: `api`, `telegram_bot`, `max_bot` (у каждого свой `Dockerfile` и `requirements.txt`).

### 5\. Запуск сервисов

В корне проекта:

```bash
docker compose up -d
```

- **`api`** — `http://0.0.0.0:8000`, эндпоинты `/api/v1/sheets/...`, проверка `X-API-Key`, монтирование `credentials.json`;
- **`telegram_bot`** — `python -m bot.main`, `depends_on: api`, `API_BASE_URL=http://api:8000`;
- **`max_bot`** — `python -m max_bot.main`, `depends_on: api`.

### 6\. Проверка логов

Чтобы убедиться, что бот запустился корректно:

```bash
docker compose logs -f
```

В логах вы увидите запуск API и ботов, а также ошибки подключения к Telegram, Max или Google Sheets.

Остановить просмотр логов: `Ctrl + C`.

### 7\. Управление контейнером

Остановить бота:

```bash
docker compose stop
```

Перезапустить бота:

```bash
docker compose restart
```

Полностью выключить и удалить контейнер (данные в таблице не трогаются):

```bash
docker compose down
```

### 8\. Обновление версии бота на сервере

Если вы изменили код бота и хотите обновить его на сервере:

1. Обновите код на сервере:
   - через `git pull`, если используете git;
   - либо повторно скопируйте обновленный проект.
2. В корне проекта выполните:

```bash
docker compose build
docker compose up -d
```

Команда `build` пересоберет образ с новыми версиями файлов, а `up -d` перезапустит контейнер с обновлённым кодом.
